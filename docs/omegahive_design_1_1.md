## 1. Assumptions

Everything downstream follows from these. If one of them is wrong, the design should be revisited at the section that cites it.

**A1 — The workload is residency, not batch.** One long-lived hive works N open-ended, related projects concurrently — and the collection itself is dynamic: projects are added, concluded, and dropped mid-flight; the hive outlives any of them. A small persistent core accumulates knowledge *across* projects; executor agents come and go, carry little state between projects, and fluctuate in number. This matches the operating reality of the hive's first user (Ben's current deployment) and his stated intent. *Consequence:* memory, evaluation, and role design target long-horizon multi-project operation; persistence is a property of a role and its state, not of a process; nothing may assume a fixed project set.

**A2 — Worker task grain spans seconds to weeks; compute is preemptible.** Assume a generous range: some tasks are a shell command or a single query; real research subagent tasks run hours to weeks ("implement a spec-based coding pipeline," "repair this formal proof using these three strategies in order"); rented compute can disappear mid-task (runpod today; **ASI:cloud** — the ASI Alliance's compute platform, the operator's intended eventual home — later). *Consequence, both ends:* at the short end, contract machinery must cost near nothing — a seconds-long task cannot carry heavyweight ceremony; at the long end, mid-task worker death is a *normal event*, so contracts carry duration, check-in, and checkpoint terms (§3.5), and recovery means "reassign from the last durable artifact," not "lose two weeks."

**A3 — Coordination writers are concurrent, out-of-process, and fallible.** Writers retry after timeouts, act on stale state, and repeat themselves — regardless of what authors the operations. In the day-1 bindings the emitting policy is an LLM turn; but emission authorship is *not* an architectural invariant: OmegaClaw could carry a hive skill in which symbolic inference decides the ops and plain code emits them through the port (a small step from the current architecture — §7 stage 2), and the design's own instruments are already deterministic non-LLM writers. *Consequence (write-path rules R1–R5 of §3.3):* these follow from concurrency and fallibility alone and are indifferent to authorship. One authorship-specific corollary: whoever supervises — an LLM planning on the board's state — must *see* refusals in its next context, whether it emitted the op itself (R4's feedback loop) or delegated emission to code (the skill must return a structured emission report). *Corollary:* every coordinator configuration presents the same writer interface, so configurations are directly comparable behind one port — with emission authorship a **declared property** of each configuration (A10).

**A4 — Independent stores drift.** Any two stores that both hold coordination state will disagree after enough failures, and reconciliation code is where correctness goes to die. *Consequence:* there is exactly **one log of record**; the task graph, the human channel, metrics, and audit are all *derived views* of it (§2). This is the largest structural difference from the 1.1 spec's bus + queue + board topology (§5.1).

**A5 — Enforcement must be structural.** Policy that lives only in prompts fails under pressure — 1.1 itself says this ("enforced by gateways and credentials, not merely by instructions inside prompts"). *Consequence:* we push the principle one level further than the spec: not only are capabilities gated by credentials, but workflow rules are **transition-legality facts**. A document task cannot *reach* the `done` state without a passed review, no matter what any agent says (§3.3, §5.6).

**A6 — The hive never grades itself.** Metrics are computed by deterministic code over raw logs; agent self-reports are never authoritative. (Shared with the 1.1 evaluation philosophy.) *Consequence:* metrics are projections of the log plus harness-side validators; the evaluation program is a separate self-contained document (the test plan) whose gates are summarized in §8.

**A7 — On bounded one-shot benchmarks, coordination rarely wins; the value case is residency.** ResearchClawBench (an SJTU-adjacent, MIT-licensed benchmark of 40 real research-reproduction tasks: given a scientific workspace, an agent must independently reach a published paper's conclusions, scored 0–100 against an expert rubric; its public archive holds 1,083 completed runs across ~27 frontier agents/models) gives the field's best picture of solo-agent research performance. We mined that archive: on 26 of 40 tasks even the *best* of ~27 runs misses 70% of the target paper — *capability-limited*; coordination is not the lever there. We classified the remaining tasks as process-sensitive where the best run scores high but cross-run spread is wide (process, not raw capability, plausibly separating runs), then read best-vs-median reports item-by-item against the rubric: only 1–2 of 10 show genuinely *coordination-shaped* misses (parallel workstreams dropped); the modal failure is one agent failing to execute one deep deliverable — a derivation not carried out, a figure not rendered — which multiplying agents does not fix. *Consequence:* we do not justify the hive by benchmark victories. Bounded benchmark comparisons are cheap supplementary checks (§8, T3 note); the primary evaluation is residency operation (A1), where the claimed value — multiplexing, cross-project memory, attention management — actually lives.

**A8 — The coordination plane must stay cheap; the compute plane is where scaling happens.** Heavy work — proof checking, browsing, experiments, inference — runs off the coordination substrate, reached as governed capabilities. *Consequence:* the log carries references, never bulk content; capacity planning targets verifier and worker services, not the coordinator.

**A9 — The agents are what they are.** OmegaClaw is a continuous-loop neural-symbolic agent: a small MeTTa program (MeTTa is Hyperon's knowledge-and-program language) running an endless sense→think→act cycle. Each cycle it assembles a fresh context — persona, skill catalog, last skill results, recent history — calls an LLM once, and treats the reply as a short list of skill invocations (bare command lines, repaired into S-expressions and evaluated in its Atomspace; **at most ~5 per cycle**). One instance = one agent: one loop, one persona, one private memory, **one active chat channel** — and the loop wakes on chat input or a timer, *not* on external events. Symbolic reasoning (NAL/PLN with tracked truth values) is invoked through those same skill calls and shapes future context; it does not act on the world directly. Adding a skill touches three places (catalog text, MeTTa equation, Python parser table), so skills are fork code, not config. OpenClaw: a mature, reliable skill-executing agent. *Consequence:* the coordinator binding works with this grain — board state reaches the agent as skill-call results (or a fork-level context injection — a stage-2 decision, §7), board operations are skills shaped to fit the agent's command parser (§3.4), and the port must serve a **mixed client population** — short-lived subprocess calls and long-lived in-process clients both occur in practice (§3.4, §10.5).

**A10 — Every non-execution agent is unproven.** Execution agents doing bounded tasks are proven technology. Everything above them is not: an LLM coordinator, an OmegaClaw coordinator, symbolic-informed planning, reflective functions — all unproven at this stage, including relative to each other. *Consequence:* the substrate is held fixed and every brain is swappable behind one interface (§3.4); the coordinator ladder starts from a scripted control; **emission authorship** (LLM turn / symbolic-decides-code-emits / script) and other capability-relevant properties (ops-per-cycle ceilings, decision latency) are *declared properties of each rung*, so the ladder compares configurations honestly; and nothing downstream (including residency) waits on any cognition proving out — residency can run on the most boring brain that passes its gates (§8).

Three ideas carry the whole design:

1. **One log.** Every coordination fact — a task created, assigned, blocked, reviewed, escalated; a refusal; a promotion — is one event in one append-only, totally-ordered log. 1.1's task graph, fast bus, audit trail, and metrics inputs are all *this log*, viewed differently. Adding a view never touches the substrate.
2. **One gateway, one policy.** An agent's only route to anything — emitting events, reading state, files, tools, models, money, the outside world — is a governed gateway under one versioned policy. Permission tiers are capability grants the gateway enforces; an agent without a grant does not hold the credential or the route. There is no ungoverned path.
3. **Roles as configuration.** A role is an emitter-authority row plus capability grants plus a model profile — data, not code, exactly as 1.1 §3.1 demands. New role, new row; the substrate does not change.

## 3. The architecture, component by component

### 3.1 The event log ("the spine")

Append-only log of coordination events; each event has a stable identity, an actor, a type, a payload of **references and small structured fields** (artifacts and tool outputs live in their own stores — A8), a server-assigned wall timestamp, and a per-run monotonic logical timestamp. Deterministic replay of the log reconstructs every view; the log *is* the audit trail (no separate hash-chained audit files — see §5.2 for when that changes). One standing invariant: **global log ≠ global capabilities** — history is readable per role policy, powers are capability grants, and **secrets never land in the log** (credentials travel as gateway-resolved handles, never as payload).

**A "run"** is the log's scoping container: one bounded span of hive operation within which total ordering, emit serialization (§3.3 R1), config/library pinning, and metrics grouping all hold. Everything in stage 0–2 operates on single-purpose runs (one board, one experiment); how runs map onto residency — one run per project vs. one long-lived hive-run with project as a task attribute — is deliberately open (§10.3) and must be settled before the residency pilot.

**Where 1.1's fast bus went.** The spec's fast bus carries high-volume working traffic — tool outputs, partial results, raw critiques. In this design that *content* never transits the spine: coordination events carry references to it, and the bulk lives in worker transcripts and the artifact store, checksummed and retained (stage 4). Inspectability is preserved — every referenced artifact is reachable from the log, and the log replay gives the timeline — but there is no separate searchable bus viewer; ad-hoc inspection is log queries plus following references, and a semantic index over transcripts is deferred until recall demonstrably needs it. This is the working-traffic half of the §5.1 divergence.

### 3.2 Projections

- **The board** — the current task graph: objectives, dependencies, status, owner, priority, blockers, provenance links. Computed by folding the log; never written directly. This is 1.1 §3.3's task graph.
- **The human channel** — 1.1 §3.4's human-legible layer. **Mechanical promotion rules** (not agent self-selection — same position as the spec) select events for humans: worker check-in and completion reports (§3.5), disagreements, blocks past threshold, outbound-action requests, money-touching requests, budget trips, loop detections, digests. Rendered through platform adapters (Telegram first; the adapter interface is platform-pluggable per 1.1 §5.5). Measured, not aspirational — the evidence: in stage-0 controlled experiments (two scenario families, a linear pipeline and a two-diamond DAG, each run at three calibrated chaos levels × 50 seeds = 300 runs, with injected worker failure, blocking, and silence), promotion recall was 1.0 (every labeled-critical situation reached the human view) and routine suppression 1.0 (no routine event promoted), at every chaos level. Full protocol in the test plan.
- **Metrics** — deterministic projections (A6). The catalog, including 1.1 §3.12's new metrics (loop coefficient, output-consumption rate, abandonment gap), is in the test plan; each metric lands at the build stage where its input data first exists (§7).
- **Audit/replay** — the log, replayed. No agent involvement required to inspect any run, live or historical.

### 3.3 The gateway and the write path

The gateway does two jobs: **policy on the way in** (who may emit what; whether a stateful transition is legal) and **projection on the way out** (what a role may see). The stores stay dumb; the policy lives in one place.

Five write-path requirements are non-negotiable — under concurrent, out-of-process, LLM-driven writers (A3) they are what "the substrate works" *means*:

- **R1 Atomic gated emits.** Evaluate-legality-then-append is one atomic unit per emit, serialized per run. Two coordinators racing to assign the same task produce exactly one acceptance and one recorded refusal.
- **R2 Idempotent event identity.** Writers supply idempotency keys; the write path returns the existing event on replay. A network retry cannot double-assign.
- **R3 One legality specification.** A single declarative structure — event type + payload discriminant → guard predicate over (board state, actor, payload) → state effect — is consulted by *both* the acceptance gate and the board fold. This kills the worst LLM-facing failure class: an event that is accepted but silently changes nothing (no error, no effect, no signal).
- **R4 Refusals are recorded values.** A refused operation returns a machine-readable code and reason *as a value* (never an exception across the process boundary) and leaves a `gateway.rejected` event in the log. Refusal rates become a free coordination metric; identical rapid-fire refusals are coalesced to cap a confused coordinator's retry loop. This is 1.1's fail-closed principle with a feedback loop attached.
- **R5 Server-set time.** The gateway assigns timestamps at insert; no client clock is trusted. All rules and metrics read logical time.

Permission tiers (1.1 §3.8, including Tier 1.5 propose-and-await-review) are expressed here: a tier is a set of capability grants; Tier 1.5 is "may emit a proposal event; the acting capability is held by the gateway until a review/acknowledgment event exists." Live tier changes (the spec's Psyche pause) are policy updates the gateway respects immediately.

### 3.4 The port: one binding surface for every brain

Coordinators and executors bind through a single narrow API (`read` → consistent board-snapshot + events + cursor; `emit(op, idempotency_key)` → `Accepted | Rejected(code, reason)`), designed for stateless short-lived clients (A9). Board state reaches the agent as a rendered view; board operations are exposed as action skills. Renderings (S-expression for OmegaClaw, prose for a plain-LLM agent) happen client-side so the port stays neutral across coordinator types — which is what makes the A10 comparison clean: rungs differ only in their declared configuration.

Four binding problems surfaced by the OmegaClaw grain (A9). Each admits two resolutions — **conform the hive to the agent, or extend the agent in the fork** — and the stage-2 spec picks per item; the default positions below are the conform-side answers, with the fork-side alternative named:

- **Op payload shape.** Default: single-string payloads, parsed inside the skill body (the agent's command parser special-cases multi-argument commands). Fork alternative: extend the parser's command table for board ops — small, but it's Python fork surface with a known three-site drift hazard.
- **Idempotency keys are client-derived from content**, not from turn counters: `key = hash(run, task, op-type, payload, …)`. A model retrying a failed op on a *later* turn must reproduce the same key, which turn-derived schemes structurally cannot do. (Supersedes the turn-counter scheme in earlier binding notes; client-side rule, no write-path impact. Exact key recipe — including how intentional repeats are distinguished from replays — is part of the port pressure test, §10.5.)
- **Within-cycle op ordering.** Default: a batch op whose client sequences the list (the agent's runtime does not guarantee evaluation order of a turn's skill calls). Fork alternative: an ordered dispatch path in the loop.
- **Refusals must outlive the turn.** Default: the binding re-surfaces unresolved rejections in the next rendered view (skill results are natively visible for one cycle only). Fork alternative: write refusals into the agent's history/memory.

**Client shape is under pressure test (§10.5).** The port was drafted for stateless, short-lived, subprocess-per-call clients; the live binding reality includes long-lived in-process clients, and a resident coordinator makes that the common case. The port must serve a *mixed population* — the pressure test determines what, if anything, in the port design changes.

### 3.5 Roles, contracts, and the elastic layer

**Persistent core** (long-lived context, cross-project state — A1): the **coordinator** (1.1's process manager: dispatch, pacing, blockers, lifecycle, iteration caps), the **planner** (1.1's research generalists: decomposition, plan revision), the **instruments** (promotion, metrics, review, provenance — deterministic emitters, not LLM agents), and **governance** (the conscience: 1.1's Psyche *function*, see §4 and §5.4). OmegaClaw instances are the natural cognitive-core candidates; persistent OpenClaw agents are an equally legitimate configuration (the spec author's own practice).

**Elastic layer:** executors bound per **task contract** — the 1.1 §3.3 contract adopted as the assignment payload, with two additions forced by A2 (marked ✚):

> objective · parent task · context bundle · allowed paths/tools · forbidden actions · model profile · done criteria · budgets (tool calls, turns, runtime, cost — malformed budget means *zero*, not unlimited) · adjudication flags (`requires_adjudication`, `patch_proposal_only`, `plan_echo`) · cancellation token · output schema (including the human-legible completion report and, for research tasks, the structured abandonment return: hypotheses tested / not tested / reason) · ✚ duration & check-in terms · ✚ checkpoint terms (what durable partial artifacts look like, so death = reassign-from-checkpoint)

Spawned-agent lifecycle (1.1 §4.4 states, including `quarantined`) maps onto board task states plus first-class attempt records.

**Check-ins and steering.** Check-in terms are bidirectional and are the heart of running long tasks under an immature coordinator. Outbound: at each sizable step boundary — or on a time interval if a step runs longer — the worker emits a structured progress report (done so far / learned / blockers / next), which the promotion rules surface in the human channel. Inbound: steering notes attached to the task (by a human, or by a stronger reviewing agent) are delivered to the worker at its next check-in. This is the software-team standup pattern: the board carries the coordination, but the work also stops to talk at known intervals — and it is deliberately how a human (or a powerful review model) re-steers an execution agent mid-task, which is what keeps large projects on track until the coordinator machinery earns that job. Two lighter alignment devices bracket the check-ins: `plan_echo` (the worker's *first* output is its restatement of the contract as a plan, reviewable before execution proceeds — adjudication applied at plan-time, cheap and effective) and the completion report (the same structure as a check-in, plus challenges and recommended follow-up, always promoted).

**Worker lifetime policy.** The long-lived "subagent that owns a whole project" in current practice is really a *plan-container*: the multi-stage plan has no home except one agent's session, so the agent lives as long as the plan. On this substrate the board is the plan-container — the planner decomposes project plans into contract-sized stages, and cross-stage state lives in checkpointed artifacts, structured returns, and the memory path. Accordingly: **the default is despawn on completion or consumption-window expiry; a long-lived contract is an explicit exception that must name the state it cannot externalize.** The honest cost of the default is re-warm overhead (a fresh worker re-reading workspace and context each stage) against the idle-context waste and drift of persistent workers — a testable bet, and we test it (test plan D4: re-warm tokens vs idle tokens vs steering-intervention counts, measured on real stage-4+ work orders) rather than argue it. **Consumption tracking** (is a worker's output actually used?) feeds the coordinator regardless of policy outcome.

### 3.6 Memory and knowledge

Layered, per 1.1 §3.5, with one governing rule: **the log is the source of coordination truth; knowledge stores are referenced from it, curated beside it.**

- **Project records and documents** — a shared, librarian-curated artifact/doc store (build stage 5), including the designated **ingestion inbox** (human-supplied specs, papers, datasets → artifacts referenced from tasks) and **publication locations** (human-readable outputs, never git-only) — see §6.5. Any TASKS.md-style file is a *view* of the board, never a second source of truth (A4).
- **Durable memory** — provenance-tagged entries (statement, scope, confidence, supersession links — the 1.1 entry shape) with an explicit review path before anything becomes durable. The *minimal* capability — provenance-tagged write and read across projects — arrives early (stage 4), because cross-project reuse is the headline residency metric (A1, §8) and must be measurable from the first pilot. **Named risk — the two memory systems:** a resident OmegaClaw also has its *private* tiers (its own history, embeddings store, Atomspace), and its native habit is to remember and recall there — invisible to the hive's reuse metric and to other agents. The mitigation is role config (durable cross-project facts routed through the hive memory skill; private memory demoted to scratch), but config is a soft control against a trained habit, so this stays on the risk list until residency data shows the routing actually happens — T5 reading zero reuse while the resident's silo grows is the failure signature to watch for. The full curator role comes later (stage 5). One open design memo (§10.1) decides the storage substrate and the private-store/hive-store relationship: there is an existing design from the OmegaClaw side, **petta-memory** (verified against the project records Jul 6): an append-only journal of typed MemoryCluster records — status derived from StatusEvents + Supersedes links, never mutated — with **four** bounded derived views (audit tail, char-budgeted prompt view, generated index view, and a **promotion-gated PLN-safe view**: raw utterances and unpromoted beliefs excluded; beliefs become PLN-eligible only via explicit PromotionEvents with bounded trust); over-budget rendering omits whole atoms, never truncates mid-atom. It is a tested standalone prototype (40 unit tests) not yet wired to OmegaClaw. Notably, its append-only + derived-status semantics are *event-sourcing in miniature* — philosophically aligned with the spine, which strengthens the convergence case the memo must evaluate: reconcile the 1.1 entry schema's mutable-looking status fields with journal semantics, and decide whether hive memory and agent memory share a substrate at all.
- **Atomspaces — stated plainly: a writable shared Atomspace would violate the one-log rule (A4), so there isn't one.** The first and most important shared Atomspace is a **direct projection of the log into Atomspace knowledge representation, immutable to agents**: a deterministic renderer turns board facts, attempt outcomes, and refusals into pre-atomized premises with harness-assigned truth values; agents query it, only the projector writes it. That makes it exactly as legitimate as the board or the human channel — a derived view — and it is the same rendering the symbolic-emitter coordinator consumes (test plan D1). Agent-private Atomspaces remain each agent's own working memory (A9). Anything an agent wants to *assert for others* is knowledge, not projection: it goes through the durable-memory path with provenance and review, or it doesn't exist. (Runtime note: spaces are process-local in the current stack, so "shared" is realized by each agent's client materializing the same projection locally — no network Atomspace service exists, and none is needed.) Divergence from 1.1 §3.5 recorded in §5.3.
- **Experiment ledger** (1.1 §3.6) — adopted as a directory convention + validator + log references (stage 4): a run record with commands, environment, seeds, artifacts, metrics, conclusion. A service only if the convention proves insufficient.
- **Semantic index** — deferred until recall demonstrably needs it.

### 3.7 The compute plane

Heavy capability lives behind the gateway as services (A8): **Lean proof-checking first** (warm pool, API access, hard resource caps — exactly 1.1 §5.3; a proof is accepted only if the checker accepts it), browser automation (read-only and side-effectful routes split by tier, per 1.1 §5.4), experiment runners, and the artifact store. We expect Lean to be the first of several verifier-shaped services (test runners, rubric scorers); the generalization is extracted when a second instance exists, not designed up front.

### 3.8 Environment, security, recovery

One generic containerized deployment: a compose profile — Postgres (the log's store), the gateway service, an OmegaClaw container, an OpenClaw gateway — secrets via environment, no host-path or hardware assumptions, runnable on a stock Linux box. The current live deployment becomes *a deployment of it*, not the definition of it. Ops floor: drain coordinators before migrating; scheduled dumps of the log store; per-run pinning of the gateway library version. Security: role-scoped credentials (an agent's container holds only its grants); no agent holds raw Docker/host-root authority; environment sanitization for subprocesses; **a human-only, agent-free recovery path (plain SSH) exists at all times** — 1.1 §5.6, adopted verbatim as an invariant. Deployment-target note: **local-ish deployments carry the program until the concept is proven** (decided Jul 6); ASI:cloud remains the intended eventual production home, its egress/secrets/preemption constraints are gathered when that move is scheduled (§9.6), and every structural security check re-runs on environment change. This whole area — deployment topology, upgrade/migration procedure, backup/restore, the recovery runbook — earns **its own detailed spec** (§10.6); this section states the invariants, not the procedures.

## 5. Divergences from 1.1, with reasons

1. **One log with derived views, not bus + queue + board as separate stores.** The spec keeps three coordination stores consistent by webhooks and discipline. Our position (A4): consistency across stores is the failure mode, not the feature — reviews of our own single-store implementation surfaced accepted-but-inert and gate-vs-fold disagreement bugs even *within* one store; three stores multiply the class. Everything the spec's queue requires — atomic claim, bounded contracts, recorded transitions, replayable audit — is met by the write path (§3.3) plus the contract schema (§3.5). The corollary for working traffic: the spine carries coordination events and references only; the fast bus's bulk content (tool outputs, partial results, raw critiques) lives in transcripts and artifact stores reachable *from* those references (§3.1). We'd propose the spec state the *requirements* rather than the store topology, so both implementations conform (§9.7).
2. **Hash-chained audit entries deferred until a trust boundary is crossed.** Chained checksums defend a file-based log against tampering. Our log lives in Postgres behind a single-writer gateway with scheduled dumps; within that boundary the chain adds little. Adopt when log material is exported or audited by parties who don't trust the store (§9.5).
3. **Shared Atomspace as immutable log-projection, not substrate.** A writable shared Atomspace would be a second coordination store (A4) and would bake in an answer to the open cognition question (A10). Instead: the canonical shared Atomspace is a read-only projection of the log (§3.6); agent-asserted knowledge goes through the reviewed durable-memory path. If symbolic consumption of the projection earns coordination authority empirically (the emitter rung, test plan D1), promote it then.
4. **Functions before residents.** 1.1 names five-plus persistent roles; our resident set starts at exactly one — the coordinator/planner (1.1's process manager function; it needs residency, and in the simplified hive it and the planner are one agent). Sys-admin work is a scheduled function over the control plane. Psyche's two operational functions — calibrating trust-but-verify (pushing back on unearned claims) and steering workers (against plan deviation and premature giving-up) — start *inside the coordinator's dispatch loop*: the abandonment return and the check-in/steering machinery (§3.5) are exactly its instruments, and a rejected too-easy abandonment is a coordination act, not a conscience act. A resident Psyche (the first operator's preference for his own hive) and a resident resource manager are operator-pack configurations, adopted when continuous context demonstrably earns its cost. This goes beyond the spec's early-operation merge allowance — a divergence, not compliance. Rationale: fewer resident agents = fewer prompts to drift, fewer contexts to fund — the spec's own "reduce idle context-window waste" argues the same direction.
5. **Verifier generalization waits for the second instance.** Lean keeps its named slot; a "verifier service category" is extracted from two working examples, not designed from zero.
6. **Enforcement via transition legality, not role behavior.** 1.1 makes the editorial gate mandatory by role design; we make it a structural fact of the state machine (A5). Evidence this matters: across all our chaos-gradient experiment runs, the false-completion rate — bad work reaching `done` — is exactly zero at every chaos level, because the transition is unreachable, not because agents behaved.
7. **`patch_proposal_only` timing.** The spec wants proposal-mode from the first implementation; there are no file-mutating workers until stage 4, so it lands there. Same principle, sequenced to its first consumer.

## 6. The operator policy pack

Everything specific to how a particular operator runs their hive is **configuration with a name**, shipped at stage 4 — not architecture, and not silently dropped. Initial contents, drawn from the first operator's live practice:

1. **Scheduled reflective tasks — two distinct loops, not one.** The **craft loop**: get better at the work you're assigned — propose/improve skills, improve the prompts driving each role, and note which models do well or badly at which task kinds (the model-qualification study records, test plan §6, are exactly this evidence); output feeds the skill lifecycle (§4, 1.1 §3.11 row) and role configs. The **Gödel loop**: work *selection* — given the hive's overall goal (stated in HIVE.md; for the first operator: self-improvement toward AGI), reflect on the work in flight and propose work that would serve the goal better; output feeds the *planner* as task/project proposals, not the skill lifecycle. The first operator runs both daily today, conflated in one reflection; the pack keeps them separate because their evidence, cadence, and consumers differ.
2. **Review cadences** — external-reviewer-model passes (a model-profile slot, vendor-free), per-worker strategic reviews.
3. **Reporting cadences** — per-contract check-in cadence defaults, periodic worker status to a channel, daily digests. (Contract check-in terms + promotion-rule + adapter config; the machinery is core (§3.5), the cadences are pack.)
4. **Sanctioned agent-to-agent conversation lanes.** The first operator reports daily OmegaClaw↔OpenClaw chats were his best source of research pivots. These lanes are preserved *deliberately*: chat-ingress channels with their own turn budgets, exempt from working-traffic loop caps, whose distilled outcomes can be promoted onto the board. Serendipity as a feature, priced as config.
5. **Ingestion and publication locations.** The *requirement* is core, not pack: a designated inbox where humans drop inputs (specs, reference papers, datasets — user inputs legitimately arrive as PDFs and documents, even though a proper hive never needs *plans* delivered that way), ingested into the artifact store and referenced from tasks; and designated human-readable output locations where results are published — git must never be the only way people can see what the hive produced. The pack holds the specific locations, formats, and watch cadences per deployment.
6. **Migration** — existing named agents keep their names as role bindings; their accumulated memory/notebooks get an explicit landing path (decided by the memory memo, §10.1).
7. **Platform adapters** — Telegram groups/allowlists as trusted-group config (1.1 §3.4's group-trust model).

## 7. The build plan

The complete path, from what exists to the full 1.1-mapped system. Each stage lands independently and names what *proves* it. (For readers of this document alone: stages 0–1 are underway; nothing later has started.)

**Stage 0 — baseline substrate (done).** A working single-process implementation of the log, gateway, board fold, promotion, metrics, and review instruments, with ~125 deterministic tests and a quarantined simulation harness (scripted stub workers, seeded randomness) used for controlled experiments. What it proved: the mechanics; promotion legibility under injected chaos (recall/suppression 1.0); the review gate's structural guarantee (false-completion 0); and a clean difficulty gradient for later comparisons. The simulation stays permanently as a deterministic regression bed — it is not the product.

**Stage 1 — the port (in progress).** Make the substrate correct under independent, out-of-process writers, per §3.3 R1–R5, and expose the §3.4 port as the one binding surface. Includes the generic containerized environment (§3.8) and, in a parallel track, a reviewed fork of the OmegaClaw runtime. (Why a fork: the upstream runtime is mid-release and must not be disturbed, and several of its boot facts are wrong for a hive resident — the entrypoint's environment scrubbing would strip the hive's database credentials; boot fetches libraries from *remote git branches*, so an unpinned restart can silently drift; there is a known dangling library import; and the embedding model must be pre-baked, never downloaded at first boot. The fork pins credential passage, vendors the boot, fixes the import, bakes the model — each change reviewed before anything is proposed upstream. Whether any Docker-level network/DNS constraint also applies is verified against the actual deploy files during this stage, not assumed.) *Proof:* two agent processes on two machines coordinate through the board over TCP with every R1–R5 property demonstrated by tests; a no-project-history operator can bring the environment up from the README and reproduce the expected board state; a transport-equivalence test shows the ported substrate produces event-identical logs to the stage-0 baseline on the same scenario.

**Stage 2 — the coordinator spike.** First LLM in the loop. Bind coordinators through the port and run the **coordinator ladder** — scripted-greedy control → plain-LLM → OmegaClaw-as-shipped (symbolic layer shapes context, LLM emits) → optionally OmegaClaw-with-symbolic-emitter (inference decides ops, code emits them through a hive skill, the LLM supervises) — on identical boards, each rung's authorship/ceiling/latency properties declared (A10). Adds per-call model telemetry and per-run model config (§4 §3.10 row). Substrate prerequisite: **join semantics for decision-forked plans** — with strict all-dependencies readiness, a doomed parallel branch blocks its join forever and no coordinator has a legal pruning move; either k-of-n readiness or a scoped prune operation must land (through the legality spec, R3) before coordination quality is measurable at all. Named binding decisions the stage-2 spec must settle (all from the OmegaClaw grain, A9/§3.4): **how board changes reach the loop** — the agent's loop wakes on chat or timer, not on board events, so an idle resident coordinator has minutes-scale decision latency by default (options: shorter wake interval / a board-notification channel adapter / a fork change waking on cursor movement), and whatever is chosen must be latency-normalized across ladder rungs; **board-view delivery** — pull (a read skill whose result the agent sees next cycle) vs. push (fork-level context injection) — these are materially different information regimes, and either way views must respect the feedback budget (truncation keeps the tail — §10.4c); **rejection persistence** and **batch-op ordering** per the §3.4 binding rules. *Proof:* an OmegaClaw coordinator drives a multi-task board end-to-end, including recovering from a structured refusal; the ladder comparison runs under a pre-registered decision rule (§8).

**Stage 3 — first executor.** A real OpenClaw executor binds through the same port and works real tasks. *Proof:* coordinator-assigned work is executed, reviewed, and completed on the board with the full trace inspectable.

**Stage 4 — contracts, lifecycle, and the operator pack** (the stage where most of 1.1's new machinery lands). Full task contracts and structured returns (§3.5, incl. abandonment fields, checkpoint terms); check-in/steering events + `plan_echo` gate (§3.5); spawn lifecycle records + consumption tracking; the worker-lifetime experiment (test plan D4) once real stage-grain work orders exist; behavioral safety floor (cancellation checks, quotas, per-turn caps, transcript checksums, rate limits); Tier 1.5 + real-outbound acknowledgment path; spend circuit breakers; experiment-ledger convention; chat ingress with group-trust + loop caps (and §6.4 carve-out); role-scoped DB credentials; `quarantined`; minimal provenance-tagged memory path (per §10.1); minimal skill-lifecycle records; **the operator policy pack (§6)**. *Proof:* the governance gate (§8, T4) passes — mandatory before any agent holds real outbound capability, credentials, or spend.

**Stage 5 — compute plane and knowledge layer.** Artifact store + librarian/curator + project records; Lean as the first verifier; notes-and-sources validation for writer workers; conscience-as-live-tier-control. The artifact store is deliberately the *first* new capability behind the gateway — it is the cheap test of whether the one-gateway/one-policy story holds when a second capability lands, caught here rather than discovered at full-hive scale. (Candidate later services — e.g. a sibling team's research-automation API — enter as unproven worker-services through the same door.) *Proof:* research-grade work products (checked proofs, ledgered experiments, gated documents) produced end-to-end with full provenance.

**Stage 6 — residency.** The hive runs as its workload demands (A1): one persistent hive, several real projects, weeks at a time — the residency pilot (§8, T5). *Proof:* the pre-registered pilot evaluation, below.

Two extensions attach after stage 3 as the workload demands them; both extend the role table, not the substrate. **Worked example:** hosting a multi-agent role protocol designed independently by a sibling team (a Main/Executor/Coder/Critic cascade currently run as prose conventions over stock agents) on the board — a usability test of whether an externally-designed pattern expresses cleanly, with its advisory critic becoming an enforced review gate. **Delegation ladder:** letting workers initiate delegation (request-a-subtask first; scoped sub-coordination later), so nesting that matters surfaces onto the board rather than hiding inside a worker's private subagents. (What a worker does *inside* its own turn — e.g. OmegaClaw's parent→child subagent dispatch work (ThreadKeeper) — is its own business within its capability grants; we support that goal while staying free to reach it differently, and this ladder is where nested delegation would surface onto the board.)
