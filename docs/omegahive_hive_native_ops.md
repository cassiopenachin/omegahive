# Hive-native operations — MVP spec (v0.2, post-panel)

**Status:** follows `omegahive_stage2_verdict.md` (the grid is closed; the pivot is to real work). This document specifies how the OmegaHive project itself becomes the hive's first workload, what must exist before that starts, and the operating principles governing granularity and steering. Amendments it implies to other docs are in §8. Readers need the design doc (`omegahive_design_1_1.md`) for substrate concepts; nothing here depends on chat context.

## 1. The decision: hive-native from now on

All project work from this point runs **on** the hive, in tiers. **Tier 1 (this spec):** the hive is the *substrate of record* — every work order is a board task; every result, report, question, and review verdict is an event carrying a pinned ref into the project workspace; executors are human-launched Claude Code sessions, with the operator as the bridge. **Tier 2:** a thin session adapter (`take task → print order → run → report`) — the worker-side binding for CLI coding agents, including both wake patterns (live tmux-driven and headless exit-and-resume) and the hooks→MCP instrumentation ladder, is specified in `omegahive_session_agents.md`. **Tier 3:** containerized autonomous executors (the hive image; qualification per `omegahive_v0b_order.md`; per-project write scoping via Landlock belongs to this tier). Tier 1 requires no new agent capability and starts generating real coordination episodes in the spine immediately — the verdict's §4 working assumption gets its falsification data from day one.

**Rationale (one paragraph of history):** four days of operating a pseudo-hive by hand surfaced three pains — process docs trapped in a hardened code repo; a human copy-paste bridge between the design partner and code sessions because questions and answers had no durable home; and plan state living in one person's memory across 2–3 parallel sessions. The coordination that mattered was never board-op arithmetic (the mechanical layer wins that) but plan maintenance under surprise. Tier 1 fixes the medium; the judgment seat is §7.

## 2. Preconditions

**Already satisfied (verify, don't build):**
1. **Watch:** CLI views exist (`report --board/--metrics/--human`, `board-view`); the read UI is an upgrade in flight, not a gate — a UI PR is already open and iteration continues through the remote window (decided Jul 10: departure does not hold for it).
2. **Tailnet:** spec confirmed (`omegahive_remote_access_spec.md`); the off-LAN test checklist must pass before departure — on the critical path, not yet run.

**To build (small, all load-bearing):**
3. **Intervene — a generic CLI emit.** The UI ack path is stage-4 by its own spec, and no generic emit command exists today (only canned demo reactors). ~Tens of lines against `Gateway.emit`; forces the actor-identity decision in §4, which is needed anyway. This is the week-1 human write path.
4. **Notify — an outbound ping on `question`/`blocked`/`escalated` events.** Without it, a blocking question raised at 2am operator-time sits until the next manual check-in; a five-week remote window degrades to batch mode. One small log-watching service, **transport: Telegram** (decided Jul 10; bot token lives in the per-service secrets scheme, deployment spec §4), outbound only, no ack path, no inbound surface. This is a precondition *because of the trip*, not in general.
5. **Backups, both stores.** Postgres backup timer (quadlet) **and a workspace mirror** — the log carries refs, never content; losing the workspace is losing the project the same as losing the log. The mirror falls out of §3's topology for free (the operator's Mac clone). After `task.reported` and the workspace go live, **re-run the deployment spec's restore drill** against the new content; note that a spine restore leaves workspace content past the restore point "phantom-ahead" — suspect until reconciled.
6. **The design partner reads the workspace directly.** Pain 2 is only closed if the design-partner side needs no human relay: the workspace clone on the operator's Mac sits in the mounted folder the design partner already reads. (This same clone is the mirror in item 5.)
7. **The project workspace (§3) and the `task.reported` build list (§4).**

## 3. The project workspace

One **light-process repo**, hosted **on Beastie** (bare repo over the already-tested SSH-over-tailnet path; no external git host — the deployment spec's egress posture stays untouched), cloned to the operator's Mac (design-partner access + mirror). Structure per project (Ben's per-project markdown convention, adapted): `projects/<name>/` containing `charter.md` (goal, constraints, current plan), `orders/` (work orders), `reports/` (dated worker reports), `questions/` (blocking questions and their answers, as files), `decisions.md` (append-only: date, decision, basis, what it supersedes), and `INDEX.md` (one line per relevant doc with its authority scope, including pointers into code repos for specs that live there). Conventions: **forward-only migration** (nothing moves out of existing repos; INDEX points backward); **refs are pinned** — every ref emitted into the spine is `path@<git-sha>`, minted at emit time, so cited content is immutable even though the working tree moves (this is what makes the workspace the artifact-store v0 without breaking replay/audit); direct pushes or same-day merges — a working medium, not a product. `decisions.md` is a human working file, not a projection of the log; the spine's relationship to it runs through pinned refs only (the cited commit is the truth at decision time). This is a deliberate, bounded exception to the one-log principle — §8 records it against design-doc A4.

## 4. One new event family: `task.reported` — and its build list

`task.reported (actor, task_id, ref, kind)` with `kind ∈ {progress, result, question, finding, reflection}` — content always by pinned ref. **Kinds are advisory labels for human and future-agent triage, not machinery**: no fold or gate branches on kind; the tier-1 consumers are the operator, the design partner, and the notifier (which pings on `question`). Control flow stays in existing lifecycle machinery: a halting question is `reported(kind=question)` plus the existing `task.blocked` transition; an answer is another report plus unblock; a discovery that should reroute *other* work is `reported(kind=finding)` plus the existing `escalate`. `reflection` is **required at task close by convention** (it is the corpus the §7 improver starts from — the data must precede its consumer; cost is one paragraph), but not schema-enforced yet. **Pre-registered smell test:** the day any consumer must *parse* report content to act, a real event type is hiding — mint it then, not before.

**Build list (the substrate work tier 1 actually requires — small, but currently zero of it exists):**
- `TaskReported` payload model + legality row (default-deny table, non-board whitelist shape) + meta-test coverage + pinned regressions, per frozen-seam discipline.
- **Actor identity for sessions:** each launched Code session operates as a registered worker — `worker.registered` at project seed, actor id stated in its work order; `blocked`/`unblocked` stay worker-owned and therefore just work. The operator and design partner get distinct actor ids with an `EMIT_AUTHORITY` row for human-tier events (no shared "operator" actor, per the UI spec's principle).
- **Idempotency:** the content+basis key rule applies unchanged (payload includes the pinned ref, so re-reporting identical content is naturally deduped).
- **`ref_report` on `TaskBlocked`** (optional, additive): structurally links "blocked" to the question that blocked it, as `ReviewPassed.ref_result` already does for reviews.
- **`decision_ref: str | None`** added now as a structured optional field on plan-mutating payloads (`TaskEscalated`, `PlanRevised`, `TaskStatusOverride`, `TaskPruned`) — recorded, never yet refused-on-missing. Structured now so the improver never regexes prose; enforcement stays deferred (§6).

## 5. Operating principles for the current loop

These govern how the operator + design partner run tier 1 today; the owner-agent inherits them in §7.

- **The granularity dial (task level).** Session lifetime is matched to the task, not defaulted: well-understood work → fine orders to cheap executors; exploratory work that resists a-priori decomposition → a coarse, long grant to a strong model **with mandatory externalization checkpoints** (stage-boundary reports and decision-log entries), so even coarse work leaves legible state. A long-lived *session* is licensed; a long-lived *ownership seat* is not (§7). During the remote window, the dial is also a defensive instrument: prefer coarse, self-contained grants sized to avoid synchronous blocks.
- **Steering is event-triggered, not polled.** The steering tier wakes on reports (question, finding, stage-complete) and budget thresholds, and reads *reports, never transcripts*. Check-in cadence stops bounding error accumulation.
- **Decision tiers:** executor discretion within the order's stated bounds → owner seat for cross-task plan mutation → human for strategy, spend, and anything irreversible. The owner seat for all current projects is the operator + design partner: the existing loop, which works and is not being replaced.

## 6. Enforcement — built lean

What the substrate can check mechanically, it does; what it cannot, conventions carry until evidence demands more. Now: `decision_ref` and `ref_report` recorded as structured fields (never refused-on-missing); refs pinned by construction at emit. Explicitly deferred: gateway refusal on missing decision refs, reflection schema-gates, decision-quality tooling, any self-optimizing decision loop. The recording is worth building regardless (recovery and the improver loop consume it); the gating is not, yet. On the record: enforcement guarantees something gets written, not that the right thing does — debugging bad decisions from recorded bases is unproven, and is not an MVP requirement.

## 7. The owner-agent ("PO") as a hive project — scope statement

Deferred and scoped, not designed here. **Ownership is a role over durable state, not a long-lived context:** the owner seat is reconstituted per wake from the workspace (charter, decisions, board state, recent reports); its input bundle is enumerable, so every plan-mutating decision can carry a harness-recorded basis. Rationale: a rehydrated owner's failures (incomplete picture) are legible and improvable at the seam; a long-lived context's failures (stale beliefs, rot, illegible state) are interior, where nothing can gate them. **Scale threshold:** small projects get no owner agent at all — if the whole project fits one healthy session and its decisions were front-loaded into the order, the project *is* a task and residual decisions go to the human tier. The owner-agent targets Ben-sized projects (days–weeks, decomposition unknown upfront), where long-lived agents are riskiest and rehydration pays; in between, one context may own a project until the project outgrows a healthy context, at which point **ownership migrates to state — never the reverse.** Opening stance when built: propose-then-human-ack on plan mutations; information diet = the workspace; first competence test = authoring work orders the operator would have written anyway. The **improver loop** rides the same rails at any tier — a periodic agent folds `reflection` reports into KB/conventions — and can start before any owner-agent exists.

## 9. Open items and named deferrals

(1) **Session execution locus: Beastie, over the tailnet** (decided Jul 10) — with the tailnet up the choice is mostly indifferent, and Beastie-resident sessions keep session state in one canonical, any-device-resumable place. (2) **Ben as a first-class author** — his PDF-plan workflow needs an ingestion path into `charter.md`/`orders/` (a converter step or agent); until then the operator proxies, which matches current practice. The granularity dial likewise presumes operator judgment; how Ben sets it himself is unresolved. (3) **Psyche and the Editorial/Communication Gate (1.1 §3.12, §3.7) are deferred, not forgotten** — `reflection` is a self-report, not independent review, and `reports/` has no evidence-quality discipline yet; both become live when residents/real outputs arrive. (4) Workspace serving through the web UI stays open — it is *not* required for pain 2 (the Mac clone closes that); if built later it needs path-traversal and snapshot-read care, not "zero new auth surface." (5) The small-project threshold stays a judgment call; revisit only if it generates disputes. (6) D4-analogue observation (rehydrated owner vs. long session with refresh skills) only if a natural candidate project appears.
